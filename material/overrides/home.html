Why Streaming Runtime?


Our Streaming Runtime aims to standardize and streamline the development and management of polyglot streaming applications by leveraging Kubernetes custom resource definitions to define application architecture. End-users will realize the following benefits:
- A standardized container runtime contract which moves much of the stream management (reading, joins, offset management and checkpointing) to runtime managed components
- Resource autoscaling to react to changes in event of throughput requirements.
- Declarative (gitops-friendly) management tooling for application management, including rollout and multi-cluster deployment
- Use of OCI images enables integration with existing supply-chain tooling. Use of function frameworks to build individual streaming processors mitigates some of the impedance mismatch between thinking in terms of stream transformations and interfacing with the streaming runtime

Our tech stack

Streaming-Lite: Applications which rely on time-ordered delivery of events for a single key, but which do not perform any cross-key operations like joins or repartitioning. These applications include windowing in some cases but not others. Typically, streaming-lite applications run as pipelined workloads in an existing scheduler framework such as TAS or Kubernetes. Examples include ETL applications and time-series analysis. An example competitor here is AWS Kinesis used with AWS Lambda.
Streaming-Fluent: Applications which perform cross-stream analysis using joins and/or repartitioning, but which use a well-defined set of constructs provided by underlying primitives. Typically, streaming-fluent applications run as standard workloads within an existing scheduler framework such as TAS or Kubernetes. Examples include Spring Cloud Data Flow and Kafka Streams.
Streaming-Expert: Applications which may perform novel or dynamic stream operations (for example, windowing based on data conditions rather than time or rows), driven by application code rather than well-defined constructs. These applications typically using a pool of ‚Äúworker nodes‚Äù, where the code and data is distributed to nodes using a framework-specific technique. Examples include Apache Spark, Beam, and Flink.
